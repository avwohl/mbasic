    Feature Completion Requirements¶

    The 5 Requirements¶

    A feature is NOT complete until ALL of these exist:

    1. Implementation - Code that makes it work
    2. Canonical List Entry - Added to docs/dev/ALL_FEATURES_CANONICAL_NAMES.txt
    3. Feature Tracking Entry - Added to docs/dev/UI_FEATURE_PARITY_TRACKING.md with status for ALL UIs
    4. Documentation - User-facing docs in docs/help/ or docs/user/
    5. Tests - At minimum, manual test procedure documented

    NEVER mark a feature as "100% complete" or "‚úÖ Done" without ALL FIVE.

    Why This Matters: Real Examples¶

    Auto-numbering Bug (2025-01)¶

    • What happened: Auto-numbering was marked "working" in Web UI but never tested
    • Result: Broken for months - JavaScript selector was wrong, function wasn't async
    • Lesson: "Implemented" ‚â† "Tested". Silent failures in UI code went undetected.

    Settings Feature Gap (2025-01)¶

    • What happened: Settings system existed but wasn't in canonical list or tracking table
    • Result: Web UI gap went unnoticed - no settings dialog implemented
    • Lesson: If not in canonical list ‚Üí not tracked ‚Üí gaps invisible

    Games Library (2025-01)¶

    • What happened: Games Library implemented but not tracked in feature parity table
    • Result: Appeared "missing" despite being functional
    • Lesson: Without tracking entry, work becomes invisible

    Process for Adding a Feature¶

    Step-by-Step Checklist¶

    1. Implement the code
    2. Write the functionality

    3. Test it works in at least one UI

    4. Add to canonical list (docs/dev/ALL_FEATURES_CANONICAL_NAMES.txt)

    5. Insert alphabetically
    6. Use consistent naming with existing features

    7. Example: "Settings Dialog" not "Settings" or "Configuration"

    8. Add to feature tracking (docs/dev/UI_FEATURE_PARITY_TRACKING.md)

    9. Find appropriate category (or create new one)
    10. Add row with feature name
    11. Document status for EACH UI:
      • CLI, Curses, TK, Web, Visual

    12. Use status format: [Implementation|Documentation|Testing]

      • [‚úÖ|‚úÖ|‚úÖ] - All three complete
      • [‚úÖ|üìö|üß™] - Implemented but needs docs/tests
      • [‚ùå|‚ùå|‚ùå] - Not implemented

    13. Write user documentation

    14. Language features ‚Üí docs/help/common/statements/ or docs/help/common/functions/
    15. UI features ‚Üí docs/help/ui/{backend}/
    16. External guides ‚Üí docs/user/

    17. Include examples and usage notes

    18. Create test procedure

    19. Automated test: Add to test suite (preferred)
    20. Manual test: Document procedure in docs/dev/ or in tracking table notes

    21. Minimum: Written steps to verify feature works

    22. Mark as complete

    Red Flags: Incomplete Features¶

    Watch for these warning signs:

    Red Flag Why It's Incomplete What's Missing
    "Code exists" but no docs Users can't discover or use feature Documentation (#4)
    "Marked as working" but no tests May be silently broken Tests (#5)
    "In tracking table" but not in canonical list Not properly registered, may be lost Canonical entry (#2)
    "Working in one UI" but not documented in all UIs Other UIs show as incomplete Tracking entry (#3)
    "Implemented" but not in tracking table Work is invisible, gaps unnoticed Tracking entry (#3)

    FAQ¶

    Q: Can I mark a feature complete if only one UI has it?¶

    No. You must document the status in ALL UIs in the tracking table. If only Web UI has it, mark Web as [‚úÖ|‚úÖ|‚úÖ] and others as [‚ùå|‚ùå|‚ùå].

    Q: What if a feature doesn't need documentation?¶

    All features need some documentation. Even internal features should have developer documentation explaining what they do and how to use them.

    Q: What counts as "testing"?¶

    Minimum: A written procedure showing how to verify the feature works (can be in tracking table notes). Better: Manual test script that can be followed step-by-step. Best: Automated test in test suite.

    Q: What if I'm adding a feature that already exists in some UIs?¶

    Follow all 5 steps: 1. Implement in your UI 2. Ensure it's in canonical list (may already be there) 3. Update tracking table with your UI's status 4. Ensure documentation covers all UIs (add your UI if needed) 5. Test and document test procedure

    Q: Can I defer documentation/testing to later?¶

    You can implement and commit code without completing all 5 requirements, BUT: - Mark the feature as incomplete in tracking table: [‚úÖ|üìö|üß™] - Create a TODO file to track remaining work - Do NOT mark as "100% complete" or [‚úÖ|‚úÖ|‚úÖ] until all 5 exist
