    Keyword Case Handling - DONE¶

    ✅ Status: COMPLETED (v1.0.122-128)

    Overview¶

    Extend the case conflict handling system to support keywords (PRINT, FOR, IF, etc.) with separate configuration from variable case handling.

    Implementation Summary¶

    Completed in v1.0.122-128 with table-based architecture (cleaner than originally proposed).

    What Was Built¶

    1. Setting Added (v1.0.122)
    2. Added keywords.case_style to src/settings_definitions.py
    3. 6 policies: force_lower, force_upper, force_capitalize, first_wins, error, preserve

    4. Default: force_lower (MBASIC 5.21 style)

    5. Token Enhancement (v1.0.122)

    6. Added original_case_keyword field to Token

    7. Lexer tracks original keyword case

    8. Table-Based Architecture (v1.0.126-127)

    9. Created CaseKeeperTable utility class (generic case-insensitive storage)
    10. Created KeywordCaseManager using CaseKeeperTable
    11. Lexer registers keywords during tokenization (lexical-level handling)
    12. Parser receives keyword_case_manager from lexer

    13. Serializer looks up display case from table

    14. Testing (v1.0.122-128)

    15. test_keyword_case_policies.py - All 5 main policies tested and working

    Key Architectural Insight¶

    Keywords work exactly like variables: - Case-insensitive lookup (normalized key) - Display case from table (first_wins/policy determines display case) - Table is single source of truth

    This is cleaner than the originally proposed runtime tracking approach!

    Proposed Implementation¶

    New Setting: keywords.case_style¶

    Add to src/settings_definitions.py:

    "keywords.case_style": SettingDefinition(
    key="keywords.case_style",
    type=SettingType.ENUM,
    default="force_lower",
    choices=["force_lower", "force_upper", "first_wins", "error", "preserve"],
    description="How to handle keyword case in source code",
    help_text="""Controls how keywords are displayed and validated.

- force_lower: Convert all keywords to lowercase (default, MBASIC 5.21 style)
- force_upper: Convert all keywords to UPPERCASE (classic BASIC style)
- first_wins: First occurrence sets case for that keyword throughout program
- error: Flag case inconsistencies as errors
- preserve: Keep each keyword exactly as typed (modern style)

Example with force_upper:
  10 print "hello"  -> 10 PRINT "hello"
  20 Print "world"  -> 20 PRINT "world"

Example with first_wins:
  10 Print "hello"  -> 10 Print "hello"
  20 PRINT "world"  -> 20 Print "world"  (uses first case)

Example with error:
  10 Print "hello"
  20 PRINT "world"  -> ERROR: Keyword case conflict!
""",
    scope=SettingScope.PROJECT,
)

    Policies Explained¶

    1. force_lower (default)
    2. All keywords normalized to lowercase
    3. Matches MBASIC 5.21 behavior

    4. PRINT, Print, print all become print

    5. force_upper

    6. All keywords normalized to uppercase
    7. Classic BASIC style

    8. PRINT, Print, print all become PRINT

    9. first_wins

    10. First occurrence of each keyword sets its case
    11. Subsequent uses displayed with first case

    12. PRINT first → all become PRINT, else Print first → all become Print

    13. error

    14. Raise error if same keyword appears with different cases
    15. Forces consistency

    16. Useful for teams with style guidelines

    17. preserve

    18. Keep each keyword exactly as typed
    19. No normalization
    20. Most flexible but can look inconsistent

    Implementation Steps¶

    1. Token Enhancement (src/tokens.py)
    2. Add original_case_keyword field to Token (similar to original_case)

    3. Lexer stores original keyword case before normalization

    4. Lexer Changes (src/lexer.py)

    5. Store original keyword case in token
    6. Keep current lowercase normalization in token.value

    7. New: token.original_case_keyword = "PRINT" (for example)

    8. Runtime Tracking (src/runtime.py)

    9. Add _keyword_case_map: Dict[str, str]
      • Maps normalized keyword → canonical case
    10. Add _keyword_case_variants: Dict[str, List[Tuple[str, int, int]]]
      • Tracks all case variants seen per keyword (for debugging/error messages)

    11. Add _check_keyword_case_conflict() method

      • Similar to _check_case_conflict() but for keywords

    12. Parser Integration (src/parser.py)

    13. Parse-time case checking if keywords.case_style == "error"
    14. Store canonical keyword case in AST nodes

    15. New field: keyword_case on statement nodes?

    16. Serializer Updates (src/position_serializer.py, src/ui/ui_helpers.py)

    17. Output keywords using canonical case from runtime

    18. Respect keywords.case_style setting

    19. Statement Node Enhancement (src/ast_nodes.py)

    20. Add original_keyword_case field to statement nodes
    21. Example: IfStatementNode.original_keyword_case = "If" (if typed as If)

    Key Differences from Variable Case Handling¶

    Aspect Variables Keywords
    Default Policy first_wins force_lower
    Historical Basis Modern enhancement Follows MBASIC 5.21
    Scope Per-variable tracking Per-keyword-type tracking
    Error Impact Runtime error Parse-time error (optional)
    Common Use Mixed case names Consistent style

    Example Programs¶

    Input with force_lower: 

    10 PRINT "Hello"
20 Print "World"
30 print "!"

    Output: 

    10 print "Hello"
20 print "World"
30 print "!"

    Input with force_upper: 

    10 print "Hello"
20 Print "World"
30 PRINT "!"

    Output: 

    10 PRINT "Hello"
20 PRINT "World"
30 PRINT "!"

    Input with first_wins: 

    10 Print "Hello"
20 PRINT "World"
30 print "!"

    Output: 

    10 Print "Hello"
20 Print "World"
30 Print "!"

    Input with error: 

    10 Print "Hello"
20 PRINT "World"

    Error: 

    Keyword case conflict at line 20: 'PRINT' vs 'Print' at line 10

    Implementation Complexity¶

    Estimated Effort: Medium (4-6 hours)

    Complexity Factors: - Similar to variable case handling (proven pattern) - Lexer already tracks token types - Serializers already handle token output - Main work: tracking and policy enforcement

    Benefits¶

    1. Style Consistency: Enforce team/project coding standards
    2. Readability: Choose case style that matches project conventions
    3. Historical Accuracy: Option to match MBASIC 5.21 or classic BASIC styles
    4. Flexibility: Developers can choose what works for them

    Risks¶

    1. Performance: Minimal - case checking is fast
    2. Complexity: Adds another setting to learn
    3. Migration: Existing code may need case adjustments if changing policies

    Priority¶

    Low-Medium - Nice to have feature, complements variable case handling

    Notes¶

    • Variable case and keyword case are independent settings
    • Keywords include: PRINT, IF, FOR, WHILE, GOSUB, RETURN, etc.
    • Does not affect string literals or comments
    • Default (force_lower) maintains backward compatibility
